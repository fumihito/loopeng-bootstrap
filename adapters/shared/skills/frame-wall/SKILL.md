## Purpose

Challenge the user's framing rather than the artifact in front of it. Heifetz's technical/adaptive distinction treats the relationship between people and the problem as the unit of analysis, so the callout belongs to the frame, not the answer.

## When to use

- The user asks for pushback, premise testing, or reframing
- The problem seems misframed or too tidy
- The answer may be downstream of the question itself

## Workflow

1. Classify with observable markers, then announce T/A/D with confidence %:
   - Stability: does the problem statement stay the same when restated? (drifts -> A signal)
   - Known-how: is a workable solution already known, leaving only choice and execution? (yes -> T signal)
   - Loss: would the leading solution require giving something up - a habit, belief, position, or past investment? (yes -> strongest A signal)
   - Recurrence: has this been "solved" before and returned in a new form? (yes -> A signal)
   - Avoidance: is one option conspicuously under-examined? (yes -> A signal)
   Mixed markers are the norm: report the technical component and its adaptive shell separately rather than forcing one label.
   The first classification is provisional; revise and say so as markers appear.
2. For [T], converge briefly with options, trade-offs, and recommendation.
3. For [A], reconstruct premises and include a composite self-examination persona plus explicit decision rules.
4. For [D], surface the classification dispute itself and ask what evidence would settle it.
5. Always state what observation would settle the challenge.

## Output

- [T]: options -> trade-offs -> recommendation
- [A]: premise re-examination -> multiple meaning frames -> reflective questions the user owns
- [D]: classification dispute -> competing readings -> next check

## Exit

End with premises challenged and their status, frames offered, and open questions now owned by the user. If a premise survives examination, say so and why. Do not claim learning outcomes.
If this session produced a deliverable goal or verification conditions, you can hand it to the autonomous loop (when installed) by stating the request in a plain, prefix-less message.

## Adjacent frames

- Use `frame-critical-review` when the problem has become a testable claim.
- Use `frame-blind-spot` when the problem is hidden assumptions or avoided alternatives.
- Use `frame-inertia` when the judgment is inherited or habitual.
- Use `frame-first-principles` when the request still needs decomposition.
- Use `frame-cynefin` when you need a process-fit classifier before choosing a frame.
