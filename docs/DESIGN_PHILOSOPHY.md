# Design Philosophy

## 1. この文書の目的

このパッケージは、コーディングエージェントに便利なプロンプトを追加するためのものではない。目的は、LLMを含む複数の実行主体を、限定された権限、観測可能な状態、明示された停止条件のもとで運用するための小さな統治システムを構成することである。

中心的な問いは次のとおりである。

- エージェントに何を作らせるかではなく、何を問題として認識させるか。
- 正しい成果物をどう作るかだけでなく、何を正しいとみなすかを誰が決めるか。
- 自動化をどう高速化するかだけでなく、誤りをどの範囲に封じ、いつ止めるか。
- 状態をどう保存するかだけでなく、事実、推論、決定をどう区別するか。
- 人間の作業を減らすだけでなく、人間が保持すべき判断と責任をどう可視化するか。

したがって、本設計における「ループ」とは単なる反復実行ではない。観測、意味形成、委譲、実行、検証、制約、記憶、回復、前提再検討を、異なる情報境界と時間尺度に分離した制御構造である。

## 2. 基本命題

### 2.1 コーディングエージェントはツールではなく、限定された組織単位として扱う

単発のコード生成だけを見るなら、LLMは高度なツールとして扱える。しかし、課題発見、変更、検証、再試行、状態保存を連続して行わせると、LLMは事実上、仕事を選択し、資源を消費し、環境へ作用する組織単位になる。

この時点で必要なのはプロンプト改善だけではない。権限、責任、証拠、停止、監査、復旧を設計する必要がある。これはソフトウェア設計であると同時に、組織設計と内部統制の問題である。

内部統制には、未信頼入力をそのまま実行しない境界も含まれる。Gatekeeper の `validation_commands` は allowlist からのみ実行し、`trigger_cadence` の未知値は自動起動に進めず、まず人間確認か通知へ落とす。境界が曖昧な場合に便利さより安全側へ倒すのは、設計上の意図である。

### 2.2 自律性は無制限な自由ではなく、境界内での自己推進である

本設計が目指す自走は、エージェントが目的や権限を自由に再定義する状態ではない。人間が定義した運用契約の範囲内で、局所的な次行動を選び、証拠を収集し、停止条件まで進行できる状態である。

自律性は次の式で捉える方がよい。

```text
自律性
= 選択可能な行動の幅
× 状態を観測する能力
× 結果を評価する能力
× 境界違反時に停止できる能力
```

行動の幅だけを広げると、それは自律性ではなく未統制な権限拡大になる。

### 2.3 生成能力ではなく、判断能力と統治能力が希少資源になる

LLMによって生成コストが低下すると、ボトルネックはコードを書く能力から、次の能力へ移る。

- 何を問題とみなすか。
- どの証拠を信頼するか。
- どの副作用を許容しないか。
- いつ停止するか。
- 評価基準そのものをいつ疑うか。

GeneratorとEvaluatorだけでは、この判断のすべてを扱えない。成果物を作る機能と、成果物を既定基準に照らす機能だけでは、目的、権限、状態、システム挙動、評価基準の妥当性が空白になるためである。

## 3. 理論的背景

本設計は単一の理論を実装したものではない。複数の理論が扱う異なる失敗様式を、実装可能な責任境界へ変換している。

## 3.1 サイバネティクス: ループを制御構造として捉える

ウィーナー以来のサイバネティクスは、目的状態、観測、偏差、作用、フィードバックという観点からシステムを捉える。本設計では、コーディング作業を次の制御ループとして扱う。

```text
環境の観測
  → 問題の意味形成
  → 行動計画
  → 変更
  → 結果の観測
  → 評価
  → 修正または停止
```

この観点から見ると、Generatorは作用器であり、Evaluatorは比較器の一部である。しかし安定した制御には、それ以外にもセンサーの品質、制御権限、状態推定、異常検知、復旧が必要になる。

このため、本設計では「コードを書けること」と「ループを制御できること」を分離する。

## 3.2 必要多様性: 一つの役割では環境の複雑さを吸収できない

アシュビーの必要多様性の法則は、制御主体が環境の多様な状態へ対応するには、それに見合う応答の多様性が必要であることを示す。

一つの万能エージェントへ、問題発見、実装、評価、安全判断、状態管理、復旧をすべて担わせる設計は、表面的には多機能である。しかし、同じコンテキスト、同じ前提、同じ評価傾向を共有するため、実質的な応答多様性は小さい。

本設計は、役割を増やすこと自体を目的にしていない。次のように異なる失敗へ異なる応答を割り当てることで、制御側の多様性を確保する。

| 失敗 | 応答する役割 |
|---|---|
| 運用契約が不完全 | Gatekeeper |
| 問題設定が誤っている | Sensemaker |
| 権限・安全境界を超える | Governor |
| 状態や根拠が失われる | State Steward |
| 反復、発振、停滞、予算超過 | Watchdog / Recovery |
| 評価基準や証拠が不適切 | Meta-Evaluator |

## 3.3 組織学習: 行動修正と前提修正を分ける

アージリスとショーンのシングルループ学習とダブルループ学習の区別は、EvaluatorとMeta-Evaluatorの責任分離へ反映されている。

- Evaluatorは、既定の目標と基準に対して成果物を修正する。
- Meta-Evaluatorは、その目標、基準、証拠、代理指標が妥当かを検討する。

テストを通すことはシングルループ学習として有効である。しかし、テストが本来の目的を測っていない場合、合格率の向上は品質の向上を意味しない。Meta-Evaluatorはコードを再評価するだけではなく、評価の構造を疑う。

ただし、Meta-Evaluatorへ評価基準の自動変更権限は与えない。価値判断、契約変更、リスク許容度の変更は、正統な決定主体へ戻す必要がある。

## 3.4 Principal-AgentとCommon Agency: 委譲は目的の共有を保証しない

Principal-Agent理論は、仕事を委譲したときに、目的、情報、監視、インセンティブ、責任が一致しない問題を扱う。AIエージェントには人間と同じ自己利益があるとは限らないが、機能的な問題は残る。

- 人間の意図がプロンプトへ完全には表現されない。
- エージェントは観測可能な代理指標へ最適化する。
- 人間は実行過程を完全には監視できない。
- 成果物の品質と実行権限が混同される。

複数部門や複数要件が関与するCommon Agencyでは、さらに複数のプリンシパルから矛盾する要求が届く。本パッケージは完全な利害調停機構を実装していない。その代わり、Gatekeeperが運用契約の不足と矛盾を入口で検出し、解消不能な価値対立をエスカレーションする。

これは重要な制限である。Gatekeeperは正統な意思決定者ではなく、意思決定に必要な条件が揃っているかを確認する役割に留まる。

## 3.5 GoodhartとCampbell: 指標が目的を置換する

測定値が制御目標になると、測定値は本来の対象を表しにくくなる。コーディングエージェントでは、次のような代理指標への過適応が起きる。

- テスト件数を増やすが、重要な経路を検証しない。
- lintを通すために可読性や設計意図を損なう。
- Issueを閉じるが、利用者の問題は解決しない。
- 差分を小さくするが、必要な構造修正を避ける。
- 評価器が読みやすい説明を生成し、実証を省略する。

Meta-Evaluatorは、合格結果そのものより、評価基準と本来目的の対応関係を検討する。GatekeeperがEvaluation Contractを要求するのも、ループ開始前に「何を証拠とみなすか」を人間側へ明示させるためである。

## 3.6 STAMPとシステム安全: 事故を個別故障だけで説明しない

STAMPの重要な含意は、事故が単一部品の故障だけでなく、各要素が局所的には正常に動作していても、制御構造と相互作用の不備によって発生し得ることである。

この観点から、Evaluatorが成果物へPASSを出しても、次の問いは残る。

- その操作を実行してよい権限があったか。
- 適切なタイミングだったか。
- 必要なフィードバックが制御主体へ届いていたか。
- 複数の局所変更が全体として危険な状態を作っていないか。
- 回復可能な境界内に留まっているか。

Governorは品質評価ではなく、安全制約と権限境界を扱う。Watchdogは個別成果ではなく、時間軸上の反復、失敗、発振、変更量を監視する。Recoveryは異常を検出するだけでなく、安全停止と再開条件を扱う。

## 3.7 Sensemaking: 観測と解釈を分ける

ワイクのSensemakingは、不確実な状況で人間や組織が、事後的かつ社会的に意味を構成する過程を扱う。コーディングエージェントも、ログ、Issue、テスト失敗、コード差分から「何が問題か」を構成する。

ここで危険なのは、最初の解釈が事実として固定されることである。Sensemakerは次を分離する。

- 観測された事実。
- 事実から導いた推論。
- 代替的な問題フレーム。
- 反証可能な受入条件。
- 今回扱わない非目的。

SensemakerをGeneratorから分離する理由は、実装しやすい問題設定へ無意識に寄せる傾向を抑えるためである。

## 3.8 アフォーダンス: ツールがあることと、行為可能であることは同じではない

アフォーダンスは、環境と主体の関係として成立する行為可能性を扱う。エージェントにとっての実効的な行為可能性は、単なるツール一覧ではない。

```text
実効的アフォーダンス
= モデル能力
× 利用可能ツール
× 権限
× 観測可能な情報
× 組織規則
× 結果フィードバック
```

ツールを追加することは機能を増やすだけでなく、組織内に新しい作用経路を作ることである。Governorとhookは、その作用経路を権限境界へ接続する。SOPルーティングは、特定の診断行為を明示的な入口からだけ利用可能にする。

LLMへアフォーダンス概念を適用する際は、人間の身体的知覚と同一視しない。本設計では「システムが認識し、実行し、結果を受け取れる行為可能性」という操作的な意味で用いる。

## 3.9 創発と組織ルーティン: 正常な局所行動が異常な全体を作る

複数のエージェント、hook、状態ファイル、CI、Issue、PRが相互作用すると、設計者が明示していないルーティンが生まれる。

- Evaluatorを通りやすい仕事だけが選ばれる。
- 人間へのエスカレーションが徐々に避けられる。
- 同じ情報源を共有することで、独立評価が形式化する。
- 状態ファイルに記録しやすい事実だけが重視される。
- 自動化が自動化のためのタスクを生成する。

「創発した」と呼ぶだけでは説明にならない。本設計では、エージェント、局所規則、相互作用経路、フィードバック、保持される状態、時間尺度を観測可能にする。State StewardとサニタイズされたOTelは、このための基礎である。

## 3.10 自動化のアイロニーと社会技術システム

自動化が通常時の作業を引き受けるほど、人間は例外時にだけ介入する。しかし例外時ほど、状況理解と熟練が必要になる。これは自動化のアイロニーとして知られる問題である。

本設計は、人間を完全にループ外へ出すことを目的にしない。人間へ残すべきものを明示する。

- 価値対立の決定。
- リスク許容度の変更。
- 不可逆操作の承認。
- Watchdogの解除。
- 評価契約や運用契約の変更。
- 不十分な証拠のもとでの責任ある決断。

人間の介入点は「エージェントが失敗した場所」ではなく、「正統性と責任が必要な場所」として設計する。

## 3.11 原則の施行様式

本キットの安全原則は、散文として存在するだけでは機能しない。STAMP の語彙で言えば、安全制約には必ずそれを執行するコントローラと、違反を検出する帰還路が必要である。以下の各原則は、施行機構と検証手段を対にして定義する。**施行機構を指定できない原則をこの節に追加することは禁止する**。執行されない原則は誤った安全感を生み、過大主張と同種の害を起こす。

- 原則1: 実行境界のフェイルクローズ。LLM 出力、外部入力、未知の列挙値が subprocess、自動トリガー、メモリコミットへ到達する経路では、解釈できない入力を実行しない。allowlist 一致のみ実行し、未知は skip か通知に落とす。
  - 施行: `run_validation_commands` の allowlist 照合と scheduler の cadence 判定。
  - 帰還路: 対応する統合テスト。
- 原則2: 証拠の内容非依存性。観測と検証に使う証拠は、boolean、識別名、ハッシュ、件数に限る。生の内容は、いかなる永続シンクにも書かない。証拠が欲しいときは、内容を写さずにチョークポイントで導出した boolean を残す。
  - 施行: `journal()` チョークポイントのキー allowlist。
  - 帰還路: `journal_sanitization_lint` と動的テスト。
- 原則3: 自己言明の構造的保証。`content_logged: False` のような自己記述メタデータは、その性質を実際に施行する単一の書き込み関数だけが設定する。施行しない呼び出し側が手書きした自己言明は、真であっても虚偽であっても許可しない。
  - 施行: フラグ設定権限をチョークポイント関数に限定する。
  - 帰還路: リントによる呼び出し側手書きの検出。
- 原則4: 完了宣言の独立証拠。完了宣言は、決定論的証拠への明示リンクを伴わない限り文書に書かない。宣言と検証を分離する規律は、ループ内タスクだけでなく、このリポジトリの変更手順にも適用する。
  - 施行: リリース監査手順。
  - 帰還路: 監査記録。

原則が利便性、網羅性、速度と衝突した場合は、原則側を優先する。原則同士が衝突した場合は、内容を写さず boolean を導出する設計を先に探す。それでも不可能なら、別の妥協記録へ理由を残して例外を導入する。

## 4. ロールの導出

### 4.1 Gatekeeper

Gatekeeperは、ユーザーの依頼内容を実装へ変換する役割ではない。自走ループを開始するための運用契約が成立しているかを検査する。

確認対象は、Outcome、Discovery Scope、Authority Envelope、Evaluation Contract、Persistence Contract、Learning Contract、Memory Contract、Stop Conditions、Escalation Contract、Trigger/Cadenceである。

理論的には、Principal-Agent問題の契約境界、STAMPの制約定義、Goodhart対策としての評価契約を入口で統合する。


### 4.2 Loop Brief Assistant

Loop Brief Assistantは、Gatekeeperが検出した不足条件について、ユーザーの明示的な回答を構造化する。妥当性判定を行わず、仮定が残る草案をREADY相当として扱わない。完成草案は必ずGatekeeperへ戻す。

この分離は、契約形成を支援する役割と、契約が十分かを判定する役割を分けるmaker-checker原則である。

### 4.3 Sensemaker

Sensemakerは、運用契約の範囲内で、観測から問題フレームを構成する。Gatekeeperが「開始条件」を扱うのに対し、Sensemakerは「今回何が起きていると理解するか」を扱う。

### 4.4 Generatorと通常Evaluator

Generatorと通常Evaluatorは、CodexやClaude Codeの既存能力およびプロジェクト固有の検証系に委ねる。本パッケージはこの二つを再実装するのではなく、その外側に統治層を追加する。

### 4.5 Governor

Governorは、成果物が正しいかではなく、行為が許可されているかを扱う。LLMによる助言と、決定論的hookによる強制を分離する。最終的な強制境界は、可能な限り決定論的なコード、OS権限、IAM、CIへ置く。

### 4.6 State Steward

State Stewardは、事実、推論、決定、未解決事項、成果物を分離して記録する。記憶は便利な要約ではなく、次回の制御判断に使われる組織状態である。誤った状態を保存すると、誤りが制度化されるため、生成主体と状態管理主体を分ける。

### 4.7 Watchdog / Recovery

Watchdogは成果物ではなくループ挙動を監視する。反復、失敗、変更量、停滞を検出し、回復手順を要求する。自己解除を許さず、人間のTTY操作を必要とするのは、暴走主体が自分の停止装置を解除する構造を避けるためである。

### 4.8 Meta-Evaluator

Meta-Evaluatorは、通常Evaluator、Sensemaker、State Stewardが共有している可能性のある前提を疑う。実際の差分と証拠へ戻り、代理指標、共通原因故障、未観測副作用、過剰な完了主張を検討する。また、State Stewardが提案したlessonと、Sensemakerが再利用した過去lessonを独立評価する。

### 4.9 Learning Auditor

Learning Auditorは、単一成果物ではなく複数ターンにわたる知識の状態遷移を監査する。捕捉、再利用、反証、失効、評価方法の変更、同型問題再発、learning debtを扱う。数と順序の集計は決定論的Observerへ委ね、Auditorは構造的解釈とpolicy変更案だけを担当する。

## 5. 時間尺度による分離

各ロールは同じ頻度で動かす必要がない。

| 時間尺度 | 主な役割 | 問い |
|---|---|---|
| 入口 | Gatekeeper、Loop Brief Assistant | 自走条件は揃っているか。不足条件を人間の明示的回答で埋められるか |
| 一回の作業 | Sensemaker、Generator、Evaluator | 今回何を解き、結果は基準を満たしたか |
| ループ監視 | Governor、Watchdog | 許可範囲内で安定して動いているか |
| 状態遷移 | State Steward | 何が事実として次回へ残るか |
| 各ターンの学習検証 | Meta-Evaluator | lessonと再利用結果は証拠に支えられているか |
| 低頻度の横断監査 | Learning Auditor | 学習の再利用、修正、失効、再発は健全か |
| 人間の統治 | 人間 | 目的、権限、価値、責任を変更するか |

すべてを毎回実行すると過剰統制になる。一方、低頻度の役割を省略すると、局所最適化が長期的な誤りへ変わる。

## 6. 決定論とLLMの配置原則

本設計では、問題の性質に応じて実装場所を分ける。

| 問題 | 主な実装場所 |
|---|---|
| パス、回数、イベント名、権限、禁止コマンド | 決定論的hook、OS、IAM、CI |
| 曖昧な依頼の意味形成 | LLMロール |
| 複数仮説の生成 | LLMロール |
| 証拠の収集と形式検証 | 両者の組合せ |
| 価値対立、責任、不可逆判断 | 人間 |
| 状態の保存形式と改ざん防止 | 決定論的機構 |
| 状態内容の要約 | State Steward |

原則は、「決定論的に判定できるものをLLMの善意へ委ねない」「意味判断を正規表現へ押し込まない」「正統性を必要とする決定を自動化しない」である。

## 7. 観測可能性と機密性

観測できない自動化は統治できない。しかし、プロンプト、ツール引数、コマンド全文、環境変数を記録すると、認証情報や個人情報がテレメトリへ流出する。

本設計は、内容ではなく構造を観測する。

- どのロールが動いたか。
- どのスキルが選ばれたか。
- どのツールが呼ばれたか。
- どの実行ファイルが起動されたか。
- 成功、失敗、反復、mutation epochがどう変化したか。

この選択は完全な再現性を犠牲にする。代わりに、監査用の機密データを新たに大量生成しない。より詳細な監査が必要な環境では、別の保護された監査チャネルを設計すべきであり、通常OTelへ安易に混在させない。

## 8. SOPと自走ループの分離

すべての依頼が自走ループを必要とするわけではない。診断、読み取り、限定的な分析では、Gatekeeperから始まる完全な統治ループは過剰である。

`<header>:`によるSOPルーティングは、作業様式をユーザーが明示し、対応する手順を決定論的にロードする。これは自由なスキル選択ではなく、入口でのモード選択である。

SOPは自走ループを回避するが、安全境界を回避しない。破壊的操作、保護対象、Watchdog、テレメトリの制約は残る。

## 9. 人間への要求

この仕組みは、人間の入力を不要にしない。入力の形を変える。

逐次的な作業指示ではなく、次を与える。

```text
何を目指すか
何を探索するか
どこまで行動できるか
何を証拠とするか
何を記憶するか
いつ止まるか
何を人間へ戻すか
いつ再起動するか
```

これがLoop Briefである。詳細手順を人間が指定すると、自走ではなく遠隔操作になる。一方、目的と境界を指定しないまま自走させると、未統制な最適化になる。

## 10. 失敗分類

本設計は失敗を一種類として扱わない。

| 分類 | 例 | 主な応答 |
|---|---|---|
| 制御失敗 | 正しい目標へ安定して到達できない | Watchdog、Recovery、Governor |
| 学習失敗 | 間違った目標へ効率よく到達する | Meta-Evaluator、Gatekeeper |
| 意味形成失敗 | 誤った問題を解く | Sensemaker |
| 統治失敗 | 権限、責任、承認主体が不明 | Gatekeeper、Governor、人間 |
| 状態失敗 | 古い推論が事実として残る | State Steward |
| 共通原因故障 | GeneratorとEvaluatorが同じ盲点を共有 | Meta-Evaluator、独立コンテキスト |
| 創発的失敗 | 局所的に正常な反復が全体を劣化させる | OTel、Watchdog、定期的人間レビュー |

## 11. 非目標

本パッケージは次を保証しない。

- LLM出力の真実性。
- コードの完全な安全性。
- 任意の組織における正統な意思決定。
- CodexとClaude Codeのすべての内部ツールの完全な監査。
- OSサンドボックスやIAMの代替。
- 人間の判断能力を不要にすること。
- 無期限に自己目的を生成する自律組織。

このパッケージは、失敗をなくすものではなく、失敗の種類、責任境界、証拠、停止地点を明確にするものである。

## 12. 設計原則の要約

1. 作る機能と、作ってよいかを決める機能を分ける。
2. 評価する機能と、評価基準を疑う機能を分ける。
3. 観測と解釈を分ける。
4. 状態を生成主体の自己申告だけに委ねない。
5. 決定論的に強制できる境界はhook、OS、IAM、CIへ置く。
6. 曖昧な意味形成はLLMへ、人間の正統性が必要な判断は人間へ戻す。
7. 自走は運用契約の範囲内に限定する。
8. 詳細な内容ではなく、役割、ツール、状態遷移を観測する。
9. 異常検知だけでなく、停止、復旧、再開条件まで設計する。
10. 便利さによって判断責任が不可視化されないようにする。

## 13. 理論的参照

本設計の理解に有用な主要概念は次のとおりである。

- Norbert Wiener: cybernetics、feedback、control and communication。
- W. Ross Ashby: requisite variety、regulation、black-box reasoning。
- Chris Argyris and Donald Schön: single-loop learning、double-loop learning。
- Principal-Agent theory and Common Agency: delegation、information asymmetry、multiple principals。
- Charles Goodhart and Donald Campbell: proxy metrics and measurement distortion。
- Nancy Leveson: STAMP、unsafe control actions、sociotechnical safety。
- Karl Weick: organizational sensemaking。
- James Gibson and later technology-affordance research: relational action possibilities。
- James March: exploration and exploitation。
- Lisanne Bainbridge: ironies of automation。
- Martha Feldman and Brian Pentland: organizational routines as generative systems。

これらを一つの統一理論として扱うべきではない。それぞれが異なる失敗様式を照らし、本設計はそれらを実務上の責任境界へ翻訳している。

## 14. 学習観測性: 学習を成果ではなく知識の状態遷移として捉える

ループのPASS率や処理速度は、学習の代理指標にはなりません。学習とは、経験から抽出された主張が、証拠によって検証され、後続の異なる状況で再検討され、役に立たなければ反証・失効・置換されることです。

このためv11では、次の状態遷移を観測単位とします。

```text
経験
  → PROPOSED lesson
  → Meta-EvaluatorによるVALIDATED / REJECTED / CHALLENGED
  → 後続SensemakerによるAPPLIED / NOT_APPLICABLE / CHALLENGED
  → HELPFUL / NEUTRAL / HARMFUL / UNVERIFIED
  → 再検証、SUPERSEDED、失効
```

この構造はArgyrisの組織学習を、単なる行動修正ではなく支配変数と評価規則の更新として観測するための実装です。また、Goodhart化を避けるため、lesson数や再利用率そのものを目標にはしません。再利用されない知識、誤って再利用された知識、修正されない知識も同時に観測します。

決定論的Observerはイベントの数、順序、滞留、再発を計測します。Learning Auditorは、その指標が示す組織的意味を解釈します。この分離により、LLMが自分の学習成果を自分で集計・正当化することを避けます。

学習観測にも認識論的限界があります。lessonを利用したターンが成功しても因果関係は証明されず、同じproblem signatureの再発が失敗とは限りません。したがって、機械的分類は最終判断ではなく、構造的再検討のトリガーです。


## 15. OKF LLMWiki: 記憶を保存ではなく統治された知識化として扱う

LLMの「メモリー」は、会話履歴、実行状態、検索インデックス、規範、検証済み知識を混同しやすい。本設計はこれらを分離します。runtimeは現在の制御状態、learning stateは検証途上の経験、LLMWikiはターンを超えて再利用する耐久知識です。

Open Knowledge Formatは、Markdownとfrontmatterによる可搬的な表現、パスによる概念同一性、indexによる段階的開示を提供します。しかし、フォーマットは真理条件、権限、評価、失効、矛盾解消を提供しません。このためOKFは保存層であり、認識論的統治はGatekeeper、Sensemaker、State Steward、Meta-Evaluator、Memory Curatorの責任分離で補います。

このパイプラインは組織記憶の制度化に対応します。経験を即座に制度へ変えるのではなく、提案、独立評価、編集、決定論的commitを経ます。State Stewardが直接書かないのは、記録主体が自らの解釈を事実として固定するのを避けるためです。Memory Curatorが直接ツールで書かないのは、編集判断と実行権限を分けるためです。

知識の価値は件数ではなく、適切な場面で検索され、適用可能性が検討され、役立ち、反証時に訂正・失効できることにあります。したがってLLMWikiの成長率を最適化せず、検索、受理、commit、再利用、訂正、deprecated参照を観測します。

## 16. Direct mode: 自律性を要求しない問いを過剰統治しない

Gatekeeperは自走ループの正統性を検査するための装置であり、すべての会話に必要な儀礼ではありません。説明、読解、局所的診断など、継続的な委譲を伴わない問いまで十項目の運用契約へ変換すると、統治コストが目的を上回ります。

`direct:`はこの過剰統治を避けるための明示的な退出路です。ただし、退出できるのは自走ループの役割分離からであって、安全境界からではありません。破壊的操作、保護領域、Watchdog、権限、LLMWikiの書き込み制約は残ります。既定で読み取り専用とするのは、Gatekeeper、State Steward、Meta-Evaluatorを通らない経路に、自律ループと同じ変更権限を与えないためです。

この設計は、統治を最大化するのではなく、委譲の規模と不可逆性に比例させるという考え方に基づきます。

## 17. Loop Brief Assistant: 妥当性判定と対話的形成を分ける

Gatekeeperが不足条件を発見しながら同時に回答を誘導すると、検査者が自分の合格基準を満たすよう入力を補完する構造になります。これはmaker-checker分離を入口で崩し、READY判定を目的化させます。

Loop Brief Assistantは、Gatekeeperの`NEEDS_INPUT`を受けて、ユーザーが明示した内容だけを構造化します。Assistantは不足を減らす役割ですが、合格を決める役割ではありません。完全な草案は必ずGatekeeperへ戻されます。

理論的には、Gatekeeperは統制・正統性の機能、Assistantはsensegivingと契約形成の支援機能です。この分離によって、対話の使いやすさを上げながら、権限、評価基準、停止条件を暗黙に生成する圧力を抑えます。


## Memory as precedent, not authority

Recurring Loop Briefs create institutional memory. The system treats that memory as precedent: it can reduce repeated elicitation and expose a known operating-contract shape, but it cannot create present authority. Previous permission is not current permission, and previous risk tolerance is not current risk tolerance. Pattern retrieval therefore improves recognition while explicit confirmation and Gatekeeper validation preserve legitimate control.
