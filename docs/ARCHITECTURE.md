# Architecture and Rejected Designs

## 1. この文書の目的

この文書は、現行構成の機能一覧ではなく、どの代替案を採用せず、なぜ現在の構造へ至ったかを記録する。

アーキテクチャは完成形ではない。CodexとClaude Codeのhook能力、スキル形式、サブエージェント制御、OTel仕様、既存リポジトリとの共存という制約の中で選択した、現時点の妥協点である。

設計判断を残す目的は、将来の変更時に「なぜ複雑なのか」を忘れて、以前に排除した危険な単純化へ戻ることを防ぐことにある。

## 2. 現行アーキテクチャの最小図

```text
User input
  |
  +-- direct: ... -------------------------------> bounded DIRECT execution
  |                                                (read-only by default)
  |
  +-- <header>: ... ------------------------------+
  |                                                |
  |                                         SOP Router
  |                                                |
  |                                    mandatory sop-<header>
  |                                                |
  |                                     bounded SOP execution
  |
  +-- ordinary request
          |
       Gatekeeper
          |
    READY / NEEDS_INPUT / REJECT
          |
          +-- NEEDS_INPUT --> Loop Brief Assistant --> Gatekeeper review
          |
        READY
          |
       Sensemaker
          |
   Generator / normal Evaluator
          |
  +-------+-----------------------------+
  |       |             |               |
Governor  State Steward Watchdog/Recovery Meta-Evaluator
  |       |             |               |
  +-------+-------------+---------------+
          |
 deterministic hooks + persistent state + sanitized OTel
          |
 deterministic learning observer
          |
 periodic `learning-audit:` → Learning Auditor
          |
 human approval, escalation, watchdog reset, policy change
```

Generatorと通常Evaluatorは、Codex、Claude Code、既存CI、テスト、プロジェクト固有スキルが担う。パッケージは、その外側へ入口、統制、状態、回復、前提評価、ターン横断の学習観測を追加する。

## 3. 採用しなかった設計

## 3.1 単一の万能エージェント

### 案

一つのエージェントに、依頼理解、実装、テスト、安全判断、状態記録、復旧、完了判定をすべて担当させる。

### 採用しなかった理由

- 同じコンテキストと前提が全工程へ伝播し、共通原因故障が起きる。
- 実装しやすい問題へフレーミングが歪む。
- 自分の成果を自分で評価し、自分の失敗を自分で記録する自己承認構造になる。
- 権限判断と品質判断が混同される。
- 失敗時に、どの機能が壊れたか区別できない。

### 文脈

短い一回限りの低リスク作業では、単一エージェントの方が効率的である。本設計が対象にするのは、反復、永続状態、変更権限、継続実行を含むループである。

## 3.2 GeneratorとEvaluatorだけの二分法

### 案

作る主体と評価する主体を分ければ十分とする。

### 採用しなかった理由

Generator/Evaluator分離は必要だが、次を扱えない。

- 依頼が自走に十分か。
- 何を問題とみなすか。
- その操作を実行してよいか。
- 何を次回の事実として残すか。
- ループが発振、停滞、自己増殖していないか。
- 評価基準そのものが目的を測っているか。

二分法は成果物品質の一部には対応するが、統治、学習、状態、回復を空白にする。

## 3.3 GatekeeperをSensemakerへ統合する

### 案

最初のエージェントが、依頼条件の確認と問題フレーミングを同時に行う。

### 採用しなかった理由

GatekeeperとSensemakerでは問いが異なる。

- Gatekeeper: この依頼は自走ループを開始できる契約になっているか。
- Sensemaker: 契約の範囲内で、今回の状況をどう理解するか。

両者を統合すると、Sensemakerが不足する権限や完了条件を暗黙に補完し、問題を解くために契約を書き換える危険がある。開始条件の妥当性と、問題フレームの妥当性は別のゲートにした。

## 3.4 Gatekeeperが不足条件を推測して自動補完する

### 案

ユーザーが権限、停止条件、評価証拠を明示していなくても、一般的な安全設定を推測してREADYにする。

### 採用しなかった理由

- 権限は利便性ではなく正統性の問題である。
- 完了条件の推測は、エージェントが目的を定義することになる。
- リスク許容度はプロジェクトごとに異なる。
- 推測した契約は、事故時の責任境界を曖昧にする。

Gatekeeperは`NEEDS_INPUT`を返し、最小限の質問を人間へ戻す。

## 3.5 すべての制御をLLMロールで実装する

### 案

Governor、Watchdog、状態ゲート、SOPルーティングをすべてプロンプトとエージェントの判断で実装する。

### 採用しなかった理由

- 禁止パス、回数上限、イベント順序、コマンド分類は決定論的に処理できる。
- LLMは自分の制約を再解釈できる。
- プロンプトインジェクションやコンテキスト欠落によって制約が消える。
- 同じモデルがGovernorを演じても、実行主体と独立した強制境界にならない。

そのため、意味判断はLLM、強制可能な境界はhookへ置く。さらにhookだけを完全な安全境界とはみなさず、OS、コンテナ、IAM、CIを外側へ残す。

補足として、`okfctl` の自己検査用 checksum と protected-path drift は改変検出のための事故防止装置であり、完全な改ざん耐性ではない。信頼境界はあくまで外側の実行環境に置く。

## 3.6 すべてを決定論的ルールで実装する

### 案

依頼の十分性、問題設定、証拠の妥当性、評価基準をJSON Schemaと正規表現だけで判定する。

### 採用しなかった理由

- Outcomeが文法的に存在しても、意味的に曖昧な場合がある。
- 権限と目的の矛盾は単純な形式検査で判定できない。
- 複数の問題フレームや反証条件は意味理解を必要とする。
- 評価指標が本来目的を測っているかは、固定ルールだけでは扱いにくい。

形式検査は出力構造と強制境界に限定し、意味判断は独立ロールへ委ねる。

## 3.7 GeneratorがState Stewardを兼任する

### 案

実装を行ったエージェントが、変更理由、証拠、次状態をそのまま記録する。

### 採用しなかった理由

- 自分に都合のよい因果説明を残しやすい。
- 失敗した試行や不確実性が省略される。
- 事実、推論、決定が混ざる。
- 状態ファイルが実装者の説得文書になる。

State Stewardは読み取り専用の独立コンテキストで、差分、テスト、hook journalから状態遷移を再構成する。

## 3.8 EvaluatorがGovernorを兼任する

### 案

テストが通り、成果物が正しければ、そのまま実行やmergeを許可する。

### 採用しなかった理由

品質合格と実行許可は異なる。

- 正しいコードでも、本番へ直接投入してよいとは限らない。
- セキュリティ上妥当でも、承認権限がない場合がある。
- ローカル変更として正しくても、外部送信や削除は別の統制を要する。

Evaluatorは「基準を満たすか」、Governorは「行為してよいか」を担当する。

## 3.9 Meta-Evaluatorが評価基準を自動更新する

### 案

Meta-Evaluatorが評価基準の欠陥を見つけたら、その場でpolicyやLoop Briefを書き換えて続行する。

### 採用しなかった理由

- 評価基準の変更は目的とリスク許容度の変更になり得る。
- 自動変更すると、失敗を回避するために合格基準を緩められる。
- Common Agency下では、どのプリンシパルの意図を優先したか不明になる。

Meta-Evaluatorは`REVISE`または`ESCALATE`を返すが、運用契約やpolicyの変更権限は持たない。

## 3.10 Watchdogが自分で解除する

### 案

Recoveryエージェントが復旧案を生成したら、自動的にWatchdogを解除して再開する。

### 採用しなかった理由

- 暴走または誤認したループが、自分の停止装置を解除できる。
- 復旧案の存在と、安全な復旧の確認は同じではない。
- 反復上限を実質的に無効化できる。

解除には人間のTTY操作を必要とする。完全無人運用が必要な環境では、別の外部オーケストレータと承認機構を設計すべきである。

## 3.11 すべての入力をGatekeeperループへ送る

### 案

診断、質問、読み取り、単発調査も含め、すべてGatekeeperから完全な自走ループを開始する。

### 採用しなかった理由

- 軽量な診断へ過剰な役割連鎖が発生する。
- 自走しない作業にTrigger/CadenceやPersistence Contractを要求するのは不自然である。
- 既知手順へ従うべき診断で、モデルが自由に作業様式を再設計する。

`<header>:`によるSOPルーティングを別入口として設けた。SOPはGatekeeperを迂回するが、安全制約を迂回しない。

## 3.12 モデルの自発的なスキル選択を信頼する

### 案

`diag:`を見たモデルが、自主的に`sop-diag`を探してロードすることを期待する。

### 採用しなかった理由

- モデルやバージョンによって選択が揺れる。
- スキルが存在しても、ロードされた証拠がない。
- 同名や類似スキルの誤選択が起こる。
- CodexとClaude Codeでスキル選択機構が異なる。

UserPromptSubmit hookがヘッダーを解析し、対応するSKILL.mdの存在、frontmatter名、配置、サイズ、UTF-8を検証し、全文をコンテキストへ注入する。

## 3.13 SOPを安全制約から完全に除外する

### 案

SOPは特別な手順なので、GovernorやWatchdogを適用しない。

### 採用しなかった理由

診断SOPでも、Bash、外部接続、ファイル変更、秘密情報へのアクセスは起こり得る。SOPは作業様式を固定する仕組みであり、権限昇格ではない。

SOPモードではループロールを起動しないが、破壊的操作、保護対象、mutation許可、Watchdog、OTelは維持する。

## 3.14 Claude Codeで`tool_input_redacted=false`相当の詳細ログを有効にする

### 案

正確なスキル名やコマンドを観測するため、Claude Codeの詳細ツールログを有効にする。

### 採用しなかった理由

詳細ログは、ツール名だけでなく、Bashコマンド全文、引数、パス、URL、検索条件、MCP入力を含み得る。認証情報、個人情報、顧客データがOTelへ流出する危険がある。

そのため、ネイティブ詳細ログは無効のままにし、hookが許可リスト方式で次だけを抽出する。

- tool.name
- command.name
- skill.name
- agent.role
- 成功、失敗、所要時間、mutation epoch

## 3.15 コマンド全文をハッシュ化して記録する

### 案

内容を平文で残さず、コマンド全文のハッシュを記録して同一操作を追跡する。

### 採用しなかった理由

- 低エントロピーな引数は辞書攻撃で推測できる。
- 秘密値の有無を相関分析できる場合がある。
- ハッシュが安定識別子となり、異なるセッションの行動を不要に結びつける。
- 監査要件に対して過剰な情報を保持する。

Watchdog内部で必要な一時的識別と、外部OTelへ出す情報を分け、外部には実行ファイル名までしか送らない。

## 3.16 CodexのネイティブOTelを自動有効化する

### 案

CodexとClaude Codeの双方で、各製品のネイティブOTelを同じように有効化する。

### 採用しなかった理由

現行のCodexネイティブOTelは、プロジェクトローカル設定の制約と、ツール結果断片を含み得るイベントの扱いにより、本パッケージの非内容ログ契約へ一致させにくい。

Codexではproject hookから共通のサニタイズ済みOTLPイベントを送る。将来、フィールド単位の抑制とプロジェクト設定が安定して提供されれば再評価する。

## 3.17 プロンプト、ツール入力、ツール出力を完全保存する

### 案

問題再現と監査のため、すべての入出力を永続保存する。

### 採用しなかった理由

- 認証情報や機密データを新たな監査データベースへ複製する。
- 保存期間、アクセス制御、削除要求、越境移転などの追加責任が発生する。
- 監査基盤自体が高価値な攻撃対象になる。
- 組織が詳細ログの存在を前提にし、最小権限や安全設計を弱める可能性がある。

通常OTelは構造的観測へ限定する。詳細監査が必要なら、別の明示的なデータ分類と保護設計を要求する。

## 3.18 Codex用とClaude用に完全に別のスキル内容を維持する

### 案

`.agents/skills`と`.claude/skills`に独立したSKILL.mdを置き、それぞれのクライアント向けに最適化する。

### 採用しなかった理由

- 同じロールの意味が時間とともに分岐する。
- 片方だけ修正され、統治契約が不一致になる。
- 両方が共有skillsシンボリックリンクを使う環境で上書き競合する。

現行設計では、必要に応じて製品別アダプタを持ちつつ、共有レイアウトではクロスプラットフォーム版を一つの実体へ配置する。製品差はagent定義とhook設定で吸収する。

## 3.19 すべてのシンボリックリンクを拒否する

### 案

リポジトリ外書き込みを防ぐため、スキルルートのシンボリックリンクを一律拒否する。

### 採用しなかった理由

CodexとClaude Codeを併用する環境では、独立コピーを許容するだけでも意味の分岐が発生する。そのため本設計では、`.agents/skills`と`.claude/skills`を共通の`{ROOT}/skills`へ向ける構成を任意の互換形ではなく正規形とした。一律拒否は正常な併用レイアウトを壊し、独立ディレクトリの許容は規律の分岐を再導入する。

代わりに、次を決定論的に検証する。

- `{ROOT}/skills`が実ディレクトリである。
- 両リンクの格納値が正確に`../skills/`である。
- リンク先が同一リポジトリ内にある。
- リンク先がディレクトリである。
- 循環していない。
- 保護領域ではない。
- 両クライアントが同じ実体を指す場合、共有スキルを一度だけ配置する。

外部リンクは引き続き拒否する。

## 3.20 `.codex`通常ファイルを常に競合として拒否する

### 案

`.codex`はディレクトリでなければならないため、通常ファイルがあればすべて停止する。

### 採用しなかった理由

既存環境では、過去のCodexプロジェクト設定を`.codex`というTOMLファイルへ置いている場合がある。内容が妥当なTOMLであれば、移行先は決定論的に`.codex/config.toml`と判断できる。

現行インストーラーは、UTF-8とTOML構文を検証し、原本をバックアップしたうえで内容を変えずに移行する。TOMLでない場合や意味が不明な場合は停止する。

## 3.21 既存設定をパッケージ版で上書きする

### 案

インストールを簡単にするため、hooks.json、settings.json、AGENTS.md、CLAUDE.md、SKILL.mdをパッケージ版で置換する。

### 採用しなかった理由

- 既存hook、権限、プロジェクト固有指示を失う。
- セキュリティ制約を弱める可能性がある。
- 同名スキルが別の目的で存在する場合、意味的破壊になる。
- アンインストールや原因追跡が困難になる。

JSONは構造マージ、Markdownは管理マーカー、`.gitignore`は不足行追加、管理ファイルはバックアップ付き更新とする。

## 3.22 競合はすべて自動的にバックアップして続行する

### 案

型が違うファイルをすべて退避し、必要な構造を作ってインストールする。

### 採用しなかった理由

退避はデータを失わないが、意味を保持するとは限らない。既存`.codex`ファイルや同名スキルに重要な設定があっても、新構成から参照されなくなれば実質的には機能喪失である。

`--conflict backup`は、人間が単純移動でよいと判断した場合だけ使う。意味的競合には`--conflict agent`を用い、LLMが実ファイルを読み、バックアップ後に意味を再統合する。

## 3.23 決定論的インストーラーが任意の意味的マージを行う

### 案

同名スキル、独自agent定義、任意の旧設定を、インストーラーが自動的に解釈して統合する。

### 採用しなかった理由

- 任意テキストの意味的同一性を安全に判定できない。
- より厳しい権限と、より広い機能のどちらを優先するかは文脈依存である。
- 設定内に機密情報があるため、外部計画へ内容を複製すべきでない。
- 誤マージは構文エラーより検出しにくい。

決定論的インストーラーは型、構文、既知レイアウト、管理マーカーを処理する。意味的競合は、ハッシュとパスだけを含む計画を生成し、リポジトリ内で動くLLMへ委ねる。

## 3.24 LLM代理インストールへ無制限の裁量を与える

### 案

LLMへ「よい感じにマージして」と依頼し、手順や検証を指定しない。

### 採用しなかった理由

- 既存ファイルを先に上書きし、あとから復元できなくなる。
- バックアップを取らない。
- 競合を解消するために権限を緩める。
- 構文が通るだけで意味的保持を確認しない。

`INSTALL_AGENT.md`は、調査、バックアップ、構造競合解消、基準導入、意味再統合、検証、diff確認、報告の順序を固定する。`sop-install`だけが明示的な変更権限を持つ。

## 3.25 hookを完全なサンドボックスとして扱う

### 案

PreToolUseで拒否できるため、OS権限、コンテナ、IAM、ブランチ保護を簡略化する。

### 採用しなかった理由

- 製品によって観測できない内部ツールがある。
- hookが呼ばれない経路や製品更新があり得る。
- Bash文字列分類は完全ではない。
- hook自体が変更、無効化、障害の対象になる。

hookは説明可能なガードレールであり、信頼境界の最外層ではない。高リスク環境では、OS、コンテナ、ネットワーク、IAM、CIを維持する。

## 3.26 無期限の自走ループ

### 案

仕事が見つかる限り、エージェントが新しい仕事を発見して永続的に実行する。

### 採用しなかった理由

- 仕事の発見自体が自己目的化する。
- 評価しやすいタスクへ偏る。
- コストと変更量が予測不能になる。
- 目的や環境が変わっても古い契約で動き続ける。

自走は一つの運用契約、予算、停止条件、trigger/cadenceに束縛する。次回実行は外部スケジューラが起動し、State Stewardの状態を読み込む。

## 3.27 PASS率を学習の指標にする

### 案

完了率、テスト成功率、Meta-EvaluatorのPASS率が上昇していれば、ループが学習していると判断する。

### 採用しなかった理由

評価器が甘くなる、対象タスクが易しくなる、失敗が観測されなくなるだけでもPASS率は上がります。これはGoodhart問題を直接招きます。v11では、知識の捕捉、後続ターンでの検討、再利用結果、反証、失効、同型問題再発を別々に観測します。

## 3.28 lesson数を学習量として最大化する

### 案

各ターンで多数のlessonを生成し、累積件数を学習の進捗とする。

### 採用しなかった理由

未検証のメモが増えるだけで、知識の品質・再利用性・失効可能性は保証されません。記録件数を目標化すると、State Stewardが細粒度のlessonを量産する誘因が生じます。lessonはMeta-Evaluatorによる検証と後続利用の観測を経て評価します。

## 3.29 LLM自身に学習履歴を集計させる

### 案

Learning Auditorがすべての過去ログを読み、再利用率や再発率も自然言語判断で計算する。

### 採用しなかった理由

計数ミス、選択的参照、コンテキスト制限、自己正当化が発生します。数・順序・期間・ID整合性は決定論的Observerで処理し、LLMは構造的解釈だけを担当します。

## 3.30 embedding類似度で同型問題とlesson再利用を自動判定する

### 案

プロンプト、Issue、ログ、lesson本文をembedding化し、意味的類似度から再発と再利用可能性を自動検出する。

### 採用しなかった理由

入力内容の外部送信または保存範囲を増やし、機密性、再現性、モデル更新による判定変化の問題を生みます。標準構成では、Sensemakerが非機密の安定`problem_signature`とlesson IDを明示し、Observerはそれだけを決定論的に追跡します。embeddingは別途セキュリティレビューを通した拡張としてのみ検討します。

## 3.31 validated lessonを自動的に恒久ルールへ昇格する

### 案

一度Meta-Evaluatorが受理したlessonを、AGENTS.md、CLAUDE.md、policy、テスト基準へ自動追加する。

### 採用しなかった理由

局所的経験が一般規則として固定され、適用範囲外で害を生む可能性があります。lessonにはapplicability、invalidation conditions、review_after_turnsを持たせ、恒久ルールへの昇格は人間の統治判断または別の明示的変更ループに委ねます。

## 3.32 Learning Auditorを毎ターン必須にする

### 案

各ターン終了時にLearning Auditorを起動し、学習健全性を毎回評価する。

### 採用しなかった理由

単一ターンでは長期傾向を判断できず、コストと遅延だけが増えます。各ターンでは決定論的観測だけを行い、Learning Auditorは一定ターン数、週次、異常発生時などの低頻度SOPとして起動します。


## 3.33 会話ログをそのままLLMWikiへ保存する

### 案

全プロンプト、応答、ツール結果をMarkdownへ追記し、後から検索可能にする。

### 採用しなかった理由

実行履歴と耐久知識が混ざり、機密情報、古い仮説、自己正当化、検索ノイズが恒久化します。runtime journalは内容を最小化し、LLMWikiには独立評価された再利用可能な知識だけを昇格させます。

## 3.34 State Stewardが直接Wikiを書く

### 案

状態を最もよく知るState Stewardに`llmwiki/`の編集権限を与える。

### 採用しなかった理由

記録と制度化が同一主体になり、推論を事実として固定しやすくなります。State Stewardはproposalを作り、Meta-Evaluatorが独立分類し、Memory Curatorが編集し、Goトランザクションが反映します。

## 3.35 Meta-Evaluatorが受理したlessonを自動的にWikiへコピーする

### 案

accepted lessonをそのままOKF conceptへ変換する。

### 採用しなかった理由

lessonはターン内学習の単位であり、概念同一性、重複、引用、authority、sensitivity、既存conceptとの関係を十分に持たない場合があります。Memory Curatorを別置し、耐久知識としての編集判断を行います。

## 3.36 Memory Curatorへ直接書き込み権限を与える

### 案

CuratorがWrite/Edit/BashでMarkdown、index、logを更新する。

### 採用しなかった理由

部分commit、index不整合、秘密情報混入、失敗時の中間状態、自己承認が起こり得ます。Curatorは完全な文書を返すだけとし、Goの標準ライブラリだけで実装した`okfctl`が全操作を一括検証して原子的に反映します。

## 3.37 ベクトルDBを記憶の正本にする

### 案

embeddingとベクトルDBだけを長期記憶とし、原文書を持たない。

### 採用しなかった理由

意味同一性、差分、引用、失効、レビュー、モデル更新時の再現性が不透明になります。OKF Markdownを正本とし、将来embeddingを加える場合も派生インデックスとして扱います。

## 3.38 OKFのすべてのYAML表現を自前パーサで受理する

### 案

外部依存なしで完全なYAMLパーサを実装する。

### 採用しなかった理由

複雑なYAML仕様を不完全に再実装すると安全性が下がります。標準構成のproducerはtop-level scalarとinline listに限定し、制約を明示します。これはOKF全体の制限ではなく、この決定論的producer profileの制限です。

## 3.39 conceptを物理削除して常に最新状態だけを残す

### 案

古くなった知識を削除し、検索結果を簡潔にする。

### 採用しなかった理由

過去の判断文脈、依存リンク、失敗した前提が失われます。既定ではdeprecatedとsupersedesを使い、Decision Logを追記します。法的・機密上の削除要求は通常のmemory promotionとは別の人間統治手続きで扱います。

## 4. 現行構成に残る妥協

## 4.1 完全な独立評価ではない

別サブエージェントを使っても、同じ基盤モデル、同じ学習データ、同じリポジトリ、同じテストを共有する。独立性は制度的に改善されるが、完全ではない。

高保証用途では、異なるモデル、独立したホールドアウト評価、外部静的解析、人間レビューを追加する必要がある。

## 4.2 Common Agencyの本格的調停は未実装

Gatekeeperは矛盾を検出できるが、複数プリンシパルの効用や権限を正統に集約するArbiterではない。部門間対立や法的責任をAIに解かせないため、意図的にエスカレーションへ留めている。

## 4.3 創発的挙動の観測は限定的

OTelは構造的情報だけを出すため、なぜ特定の行動を選んだかを完全には再構成できない。内容ログを排除したことによる意図的な制約である。

## 4.4 製品hookの能力差が残る

CodexとClaude Codeはイベント名、ツール可視性、サブエージェント制御、設定位置が同一ではない。共通hookランナーは意味を揃えるが、完全な機能対称性は保証しない。

## 4.5 人間の能力維持を機械的には保証できない

State StewardやMeta-Evaluatorが説明を残しても、人間が実際に読み、判断能力を維持するとは限らない。自動化のアイロニーへの対策は、技術だけでなく、レビュー頻度、訓練、責任設計を必要とする。

## 4.6 原則の施行は完全ではない

ここまでの設計は、原則をチョークポイントとリントへ落とし込むことで、散文だけの統治よりは強い。しかし、チョークポイント自体の誤実装、リントの抜け、自然言語ドキュメントの整合、外部ツールや他言語実装への波及までは完全には埋められない。したがって、原則はコード、テスト、静的検査、そして人間レビューの組み合わせで維持する前提であり、単一の機構に完全性を期待しない。

## 5. 将来再評価すべき条件

次の変化があれば、採用しなかった設計を再評価できる。

- CodexとClaude Codeが、プロジェクトhookで全ツールを統一的に観測・拒否できる。
- ネイティブOTelにフィールド単位の厳密な除外設定が提供される。
- スキルロードの証明と、サブエージェント権限の共通仕様が提供される。
- policy-as-codeエンジンと製品hookが直接統合される。
- 暗号学的に保護された詳細監査ログが、通常テレメトリと分離して利用できる。
- 複数プリンシパルの承認ワークフローを扱う外部ガバナンス層が導入される。

再評価時にも、便利になったことと、安全境界が不要になったことを混同しない。

## 6. 変更判断の基準

今後アーキテクチャを変更する場合、少なくとも次を確認する。

1. どの失敗様式を減らす変更か。
2. 新たに誰へ権限が移るか。
3. その権限を誰が停止できるか。
4. 事実、推論、決定の境界が崩れないか。
5. 独立評価が形式だけにならないか。
6. 機密情報を新たに記録しないか。
7. 既存プロジェクト固有設定を保持できるか。
8. CodexとClaude Codeで意味が同じか。
9. hook障害時に外側の安全境界が残るか。
10. 人間の正統な判断を、利便性のために暗黙委譲していないか。

## 7. 結論

現行アーキテクチャの複雑さは、エージェント数を増やしたいからではない。異なる失敗を一つの主体へ集約すると、失敗が不可視になり、自己承認と自己解除が起きるためである。

同時に、すべてを分離すればよいわけでもない。低リスクな単発作業には軽量なSOPを使い、決定論的に処理できる境界はhookへ置き、価値と責任を伴う判断は人間へ戻す。

このパッケージの設計原則は、最大自動化ではなく、境界の明示された自動化である。

## 3.40 `direct:`を通常ループの省略記法にする

### 案

`direct:`をGatekeeperだけ省略し、Sensemaker、変更、State Steward、Meta-Evaluatorへそのまま進める。

### 採用しなかった理由

Gatekeeperを外した状態で変更権限だけを維持すると、目的、権限、停止条件、評価契約が未定義のまま自律実行できてしまいます。Direct modeは非自律・既定読み取り専用として隔離し、変更を許可する場合も明示的なローカルポリシー変更を要求します。

## 3.41 `direct:`を汎用SOPとして実装する

### 案

`sop-direct`を作り、通常のヘッダールーターで処理する。

### 採用しなかった理由

Direct modeは特定の手順を強制するSOPではなく、Gatekeeperを通さない一回的対話というルーティング種別です。存在しないスキルへの依存や、任意のSOP本文による意味変更を避けるため、予約ヘッダーとしてhookで直接処理します。

## 3.42 Gatekeeper自身が不足項目を聞き続ける

### 案

`NEEDS_INPUT`の質問をGatekeeperが直接ユーザーへ返し、次ターンもGatekeeperが草案形成と検査を続ける。

### 採用しなかった理由

検査者が入力形成を兼ねると、質問の目的が条件の明確化からREADY獲得へ変質しやすくなります。Loop Brief Assistantへ対話を分離し、完成草案をGatekeeperが再検査する構造にしました。

## 3.43 Loop Brief Assistantが不足条件を合理的に推定する

### 案

一般的な安全慣行やリポジトリの既存設定から、不足する権限、評価条件、停止条件を自動補完する。

### 採用しなかった理由

推定は提案としては有用でも、委譲契約の事実にはなりません。特にauthority、memory、escalationは正統な決定主体の明示が必要です。Assistantは正規化と重複除去だけを行い、仮定が残る草案を`READY_FOR_REVIEW`にできません。


## Rejected: silently replaying the nearest prior Loop Brief

A nearest-match replay would be convenient but would transfer old authority, stop conditions, escalation ownership, and memory policy into a new context. Similar task descriptions do not imply identical risk or authorization. The adopted design retrieves reviewed patterns, presents field-level suggestions, requires explicit confirmation for every applied field, and sends the result back through Gatekeeper.

## Rejected: letting Loop Brief Assistant write LLMWiki directly

The Assistant is optimized for dialogue completion and is exposed to the user's current wording. Direct write access would collapse elicitation, abstraction, evaluation, and persistence into one role. Instead, the Assistant proposes, Gatekeeper independently assesses, Brief Pattern Curator abstracts and deduplicates, and Go performs the transaction.
